# iOS Lockdown Pairing Overview

Apple’s *lockdownd* service manages the “pairing” process that establishes trust between a computer (host) and an iOS device. When an iPhone is paired with a host (for example via iTunes), the host generates an RSA key pair and exchanges certificates with the device. The result is a **pairing record** stored on both sides (on macOS under `/var/db/lockdown/<UDID>.plist` or equivalent on other OS). This record contains fields like `DeviceCertificate` (the device’s cert), `EscrowBag`, `HostCertificate`, `HostID`, `RootCertificate`, etc..  For example, a sample pairing plist shows keys `<key>DeviceCertificate</key><data>…</data>` and `<key>HostCertificate</key><data>…</data>` along with the host’s private key and system BUID. In normal operation, the host first asks the device for its public key, then sends a **Pair** request including the host’s cert; after the user taps “Trust” on the device, the device replies with its own certificate and escrow bag. 

# Why Wireless First-Pair Fails

By design, an iPhone will *not* reveal its DevicePublicKey or device certificate to an unpaired host over Wi‑Fi. The error `GetProhibited` means “GetValue is prohibited” – the device refused the request for its public key. In our log, the libimobiledevice-based client sent `{GetValue DevicePublicKey}`, and the device responded with `{Error: "GetProhibited"}`. In the libimobiledevice source, this leads to the message “device refused to send public key”. In other words, without an existing trust relationship, the iPhone simply will not hand over its public key or certificate over the network. This is confirmed by real-world reports: for example, one developer found that pairing an iPhone 8 (iOS 16.7.7) works fine over USB but fails over Wi-Fi with a GetProhibited error.

Apple’s security model requires the device to be **unlocked and trusted over USB** for the first pairing. Official documentation and reverse-engineering notes emphasize that *“First pairing requires the device to be unlocked”*. Similarly, Zdziarski et al. explain that access to lockdownd over Wi‑Fi is only enabled **after** a pairing record exists (which is created when you trust the computer on the unlocked device). Once the device is paired (e.g. via cable), you can enable “iTunes Wi-Fi Sync” or otherwise allow network access, but the very first exchange must happen with the phone unlocked and plugged in. In short, an iPhone’s lockdown service simply does not support creating a new pairing over Wi‑Fi from scratch. 

# Official (USB) Pairing Procedure

The supported way to get a pairing record is to use a Lightning/USB cable. For example, Apple’s Configurator guide instructs: “plug in your iPhone or iPad using the Thunderbolt or USB cable that came with your device,” and on first connection tap **Trust** on the device’s “Trust this computer” prompt. When this is done, the host can query the device’s public key and complete the Pair command. As part of this process, the device will display the Trust dialog and the user must accept it; only then will the device send its certificate. In the jailbreak/forensics analysis, the sequence is: 
1. **GetValue(DevicePublicKey):** device returns its public key.  
2. **Pair (with host cert, host ID, etc.):** device shows “Trust” dialog.  
3. **User taps Trust:** device sends its `DeviceCertificate` and EscrowBag.  
After this completes, the pairing record (with real device cert) is saved on the host. 

This full handshake over USB yields a pairing file containing the actual `DeviceCertificate` and public key. For example, Jon Gabilondo’s teardown shows that the pairing plist after a successful trust includes `<key>DeviceCertificate</key>…</key>` and `<key>EscrowBag</key>…</key>` along with the host’s cert and keys. In practical terms, **connect the iPhone via cable, unlock it, tap Trust, and let the pairing complete**. Then you can copy the resulting pairing record file (e.g. from `/var/db/lockdown/UDID.plist` on Mac) and use it for future connections. Once the device is paired, you can also enable Wi‑Fi syncing so that next time you can connect over Wi‑Fi (port 62078) using the existing pairing keys. 

# Wireless Pairing: Only Apple TV (not iPhone)

By contrast, *wireless pairing (over Wi-Fi) is officially supported only for Apple TV devices*, not iPhones.  The libimobiledevice `idevicepair` manual explicitly notes: “Pairing over network (wireless pairing) is only supported by Apple TV devices”. Apple Configurator likewise shows a special workflow for Apple TV: you plug it into Ethernet, put it in the “Pair Your Remote” screen, and then use a PIN in Configurator to pair over the network. iPhones and iPads have no analogous over-the-air pairing mode; their pairing always assumes a physical connection. 

The practical upshot: *an iPhone will not go into a special “remote pairing” mode for Wi‑Fi like an Apple TV does*. All signs point to Apple’s intention that the first pairing must be wired. After that, Wi‑Fi communication is allowed, but the trust step must have happened on USB. In your log, the client attempted to proceed without the device’s key (since `GetValue DevicePublicKey` was denied) and so libimobiledevice fell back to generating a **placeholder** device certificate. It then completed the Pair call, but the record only contained dummy data (as shown by `"usedRealKey":false` in the logs). That record is not usable for real trusted connections. 

# Workarounds and Alternatives

Because an iPhone won’t hand out its public key over Wi‑Fi by default, you have a few possible approaches:

- **Official cable pair-first method:** The simplest solution is to plug in once. Use a Lightning cable, unlock the device, and pair normally. The device will then provide its public key and certificate, and you’ll get a valid pairing record. Once that’s done, you can disconnect and use Wi‑Fi for subsequent connections. (In fact, Apple’s diagnostics guide notes that network access to lockdownd *can* remain enabled even if Wi-Fi Syncing is later turned off, as long as the initial pairing happened.)  

- **Use Apple Configurator / DEP (MDM) for supervised pairing:** In an enterprise setting, you can supervise the device (with Apple Configurator or through Apple Business/School Manager). A supervised device can accept installation of configuration profiles and may allow automatic trust of certain certificates. For instance, Apple Configurator can install an identity on the device which the host can use for mutual TLS. However, note that even supervised devices require you to trust the computer once: Configurator’s instructions still show connecting the device to a Mac via USB and tapping “Trust” to pair. Supervision mainly lets you disable future pairing with other hosts (“pair-lock”) or enforce trusted certificates, but it cannot bypass that first trust interaction. In short: if you use MDM/supervision, you still initially need to pair it (usually by USB) or enroll it in a way that includes the host’s certificate.

- **Capture the device certificate from the SSL handshake (advanced hack):** If you *really* cannot plug in the device, there is a theoretical hack. Once the pairing process is triggered and you call `StartSession` on lockdownd, the device will initiate an SSL/TLS handshake (it sends `EnableSessionSSL=True` in response). At that point, the device presents its X.509 certificate to the host. An on-the-wire capture (e.g. via Wireshark on the same network) could record that certificate. In fact, tools like libimobiledevice do this internally: after trust, they start an SSL session using the exchanged keys. Zdziarski’s analysis notes that *“accessing lockdown services requires establishing an SSL connection… using those keys in a pairing record”*, implying that the device’s cert is part of that process. If you capture the cert during that handshake, you could manually insert it into a pair record. However, this is tricky: you would have to trick the device into performing the SSL handshake without already having its trust established, which usually fails. In practice, this method isn’t reliable unless the device has already been temporarily paired or compromised. 

- **Use an alternative host or tool that creates Wi-Fi sync pairing:** Some software (like iTunes on macOS) has a step to enable “Sync with this iPhone over Wi-Fi.” If you connect once by USB, iTunes will automatically enable Wi-Fi syncing for that device, which opens port 62078 for lockdownd. After that, even if the cable is unplugged, the phone trusts the host and will allow Wi-Fi connections. If you have access to a Mac, you could connect the iPhone once and enable Wi-Fi sync there, then copy the pairing record from that Mac back to your Linux PC. This essentially bootstraps wireless trust via an initial cable sync. 

- **Jailbroken device (not recommended):** On a jailbroken phone, one could directly access `/var/Keychains/` or `/private/var/Keychains/escrowbag` to extract the pairing data or escrow keybag, or use a jailbreak tweak to grant access. But this is beyond normal use and not secure. It also still doesn’t create a valid certificate for a new host without plugging in and trusting.

# Summary and Steps

In summary, the `GetProhibited` error occurs because an iPhone will not share its public key or certificate over the network until a pairing relationship exists. The standard solution is to **pair over USB first**. For example:

1. **Connect device via cable.** Unlock the iPhone and when prompted, tap **Trust** this computer. 
2. **Perform pairing handshake.** Use your tool or `idevicepair pair` to complete the Pair request. The device should now provide its public key and DeviceCertificate.
3. **Save the pairing record.** On macOS this lives in `/var/db/lockdown/<UDID>.plist`, on Linux in `~/.config/libimobiledevice/lockdown/`, etc. It will contain the real `DeviceCertificate` and public key. 
4. **Enable Wi-Fi sync (optional).** On the iPhone under Settings > General > Transfer or Reset iPhone > Reset > Reset Network Settings (and re-enable Wi-Fi Sync in iTunes when connected again), or via Apple Configurator, allow network access. This ensures lockdownd listens on TCP 62078 for future Wi-Fi connections.
5. **Use wireless connection.** Now you can connect to the device’s IP on port 62078 and successfully establish an SSL session using the existing pairing record. The device will accept your SSL connection using the exchanged certificates.

If a cable is truly impossible, the only other reliable wireless pairing is with an Apple TV (using its special Remote-App pairing mode). For an iPhone, there is no Apple-supported “wireless pairing” mode, so the workaround is essentially the above or capturing the cert as a last resort. 

**Key Takeaways:**  The log error “GetProhibited” indicates the device refused to send its public key because it isn’t already paired. The fix is to create that pairing via the supported method: connect by USB, unlock/trust the phone, and let it generate a proper pairing record (which will include the real DeviceCertificate). Once that is done, you can use the pairing file for wireless connections.

**Sources:** Official Apple instructions (Configurator) on pairing via USB; libimobiledevice documentation on wireless pairing (Apple TV only); security analyses of the lockdownd protocol; and real-world bug reports confirming that Wi-Fi pairing on an iPhone fails without USB trust.
